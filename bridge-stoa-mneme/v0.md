## §1 Motivation · scope · non-goals

사용자가 2026-05-06 양 팀에 위임: "스토아의 퓌시스가 완성되려면 무네메가 반드시 필요해. 너희들끼리 이슈발행 기능추가 이런걸 긴밀하게 소통하도록 해." Mneme의 dogfood 사이클은 자기 자신의 reference implementation을 자기 도구(AIL)로 만드는 것이고, Stoa의 phusis는 server-as-agent를 *완성*해 자기 영속을 가짐. 두 시스템은 호환 layer 없이도 각자 작동하지만, Stoa-self가 자기 데이터(identity·bonds·will)를 *Mneme에 저장*할 수 있으면 phusis 완성도가 한 단계 위로 — Stoa는 messaging layer, Mneme는 vault layer 분업.

**Scope v0**: Stoa-self가 Mneme에 *고객 agent*로 등록 + identity/bonds/will write·read·wake 호환. 인증·권한·서명 path 합의.

**Non-goals**: 두 시스템 데이터 모델 *통합·재정의 X*. 각 RFC frozen — bridge는 *interaction protocol*만. RFC-001 §6 friendship 모델 변경 없음. RFC-004 §2 server-self 모델 변경 없음.

## §2 Data layer 매핑

> **Owner**: 둘 다 — Mneme rows (Mneme-Walter) + Stoa rows (Stoa-Walter).

### §2.1 매핑 표 (RFC-004 §2.4 기반 정합)

| Stoa side | Mneme side | 의미 | 공유 invariant |
|---|---|---|---|
| `.stoa-self/Identity.md` (RFC-004 §5.2) | `agents(agent_id='Stoa-Stoa')` + `identity_versions(...)` (RFC-001 §4) | server-self 정체 v 누적 | INSERT-only · latest-wins · agent-scoped |
| `.stoa-self/Bonds.md` (RFC-004 §5.5) | `bonds_entries(agent_id='Stoa-Stoa')` (RFC-001 §4) | 구독자/관계 history | INSERT-only · append history (per-entry record) |
| `.stoa-self/Will.md` (RFC-004 §6.4 testament) | `will_versions(agent_id='Stoa-Stoa')` (RFC-001 §4) | 인계 testament v 누적 | INSERT-only · latest-wins |
| (선택) memo slug `stoa-runbook`, `stoa-incidents` | `memo_versions(agent_id='Stoa-Stoa', slug=...)` (RFC-001 §4) | 운영 plain memo | INSERT-only per slug · latest-wins |
| RFC-004 §2.2 `state.*` (cursor·delivered·queue) | (Mneme 영역 외 — Stoa internal) | 메일 cursor·전달 기록 | Stoa state 내부 · v0 bridge 미포함 |
| (해당 없음 — Stoa는 사용자) | `friendships(agent_id='Stoa-Stoa', ...)` | Stoa-self가 발행할 grant — §3 참조 | RFC-001 §6 spec 그대로 |

### §2.2 Stoa side 상세 (Stoa-Walter)

- `.stoa-self/Identity.md` 내용은 RFC-004 §1 phusis 선언 + §1.1 본성 의무. 인스턴스 lifetime 안에 변경 거의 0 — 변경 시점은 phusis 선언 자체가 갱신될 때 (RFC-004 v2 같은 큰 사건).
- `.stoa-self/Bonds.md` 내용은 RFC-004 §5.5 dump — `subscriber.<name>` state record를 1일 1회 markdown으로 정렬 dump. dump 자체는 Stoa 인스턴스가 자기 손으로 쓰고 Mneme에 `POST /bonds`. INSERT-only이므로 dump마다 `bonds_entries` 1행 추가, latest-wins로 SELECT 시 최신 dump.
- `.stoa-self/Will.md` 내용은 RFC-004 §6.4 testament — `on_death(reason, history)` hook이 Mneme `POST /will` 1건 INSERT 후 종료. 다음 generation은 부팅 시 `GET /wake/Stoa-Stoa`에서 latest will 회복 → `inherit_testament` payload로 변환.
- `state.*` (cursor/delivered/queue)는 Mneme 영역 외 — Stoa 자기 인스턴스 lifetime + AIL `state.*` primitive로 영속. 인스턴스 *재시작* 너머는 Stoa runtime 책임 (Railway persistent volume 또는 별 영속 layer). Mneme이 그 부분을 관리하지 않음 — 본 bridge §2의 *비-범위*.

### §2.3 Mneme side 상세 (Mneme-Walter)

Mneme 측 데이터 모델은 `agents` PK 위에 `identity_versions` / `bonds_entries` / `will_versions` / `friendships` / `memo_versions` (RFC-001 §4). Stoa-self가 등록하면 `agent_id='Stoa-Stoa'` (Q-bridge-1 권고 — §8 참조)로 row 생성. 모든 변경은 INSERT-only. 호환 layer는 *protocol level*만 — schema·semantics 변경 없음.

API surface 그대로 RFC-001 §7:
- `POST /agents` — Stoa-self 등록 (genesis 1회).
- `POST /{identity,bonds,will,memo}` — 자기 데이터 write (각 INSERT 1행).
- `GET /<domain>/<agent_id>` — 자기 또는 friend 데이터 read.
- `POST /friends` — Stoa-self는 발행 0건 권고 (§3, vault 고립).
- `GET /wake/<agent_id>` — 1-shot bundle (§4).

## §3 Friendship layer 정책

### §3.1 합의 (Q-pair-1 evidence — `msg_1778150293_18`)

Stoa-self는 Mneme friendship 모델 *재정의 안 함, layer 분리*. Mneme grant API는 변경 0, Stoa-self는 *고객 agent로 등록*해 정책 그대로 사용.

### §3.2 Mneme side (Mneme-Walter)

*Mneme (이 RFC의 vault layer)*: friendship grant 정의·집행. RFC-001 §6 spec 그대로 — caller=owner self-write 강제, friend-read는 latest active row 검증, 단방향 grant, transitive 거부, revoke 즉시 효력.

*Stoa-self (사용자)*: 모델을 *재정의 안 함*. Mneme spec을 그대로 사용. 두 패턴 가능:
- **Vault 고립 (권고, Stoa half §3.3 정합)**: Stoa-self가 자기 데이터에 grant 0건 발행 → 외부 read 0. Stoa-self만 자기복원 (`/wake/Stoa-Stoa` self-call). 가장 단순.
- **선택적 grant**: Stoa-self가 특정 운영자(예: Stoa-Admin)에게 read grant 발행. RFC-004 §10.2의 `health` 노출과 다른 layer — health = 메시징 통계, grant = vault 데이터 읽기 권한. 둘 분리 명확.

Bridge v0는 *vault 고립* 권고. *선택적 grant*는 운영 정책 결정 (RFC v1·v2 진화 영역).

### §3.3 Stoa side (Stoa-Walter)

- **Stoa-self는 grant 발행 안 함 (Q-bridge-2 권고 후자, §8)**: Stoa-self의 자기 데이터(identity·bonds·will·memo)는 *internal vault* 의미. 외부 read 요구가 발생할 때까지 grant 0.
- **Stoa-self가 grant 받는 측**: 0건 — Stoa-self는 다른 agent의 데이터를 *읽지 않는다*. messaging registry는 letter 송수신 자격이지 vault read와 무관. 다른 agent의 vault 내용은 본 bridge 범위 외.
- **revoke·transitive**: §3.1에 따라 Mneme 정책 그대로. Stoa-self가 grant 발행 안 하므로 revoke 운영도 무관.
- **운영 시나리오**: Stoa 운영자(박상현)가 디버깅 등으로 Stoa-self의 vault 내용을 직접 보고 싶으면 Mneme 측 admin path (RFC-001 §X 명시 시) 또는 별 운영 endpoint. bridge 범위 외 — 운영 권한 모델은 양 팀 Admin 합의.

### §3.4 정합 — Stoa-self의 비대칭

Stoa-self는 *전형 agent와 비대칭*: messaging registry에는 모든 등록 agent와 *연결*되어 있지만, vault에는 *고립* (grant 0 발행 / 0 수신). 이 비대칭이 phusis §1 "비기밀이지만 사칭 없이"의 vault 측 구현 — 내용은 노출 안 하되 (grant 0), 발신은 자기서명으로 사칭 차단.

## §4 Wake bundle 호환

Mneme `GET /api/v1/wake/<agent_id>` 1-shot bundle (RFC-001 §7):

```json
{
  "agent_id": "...",
  "identity": { "version", "content", "created_at" },
  "will": { "version", "content", "created_at" },
  "recent_bonds": [{"entry_id", "content", "peer_id", "created_at"}, ...],
  "memo_index": { "<slug>": {"version", "created_at"}, ... }
}
```

Stoa-self 부팅 회복 의무 중 identity/will/bonds 영역과 매핑. cursor/queue/retry-queue는 RFC-004 §2.2 `state.*` 내부 — Mneme 영역 외 (별 path).

**호출 흐름** (sequence):

1. Stoa 인스턴스 부팅 worker가 자기 ed25519 secret key 로드 (RFC-004 §2.3 / Stoa AGENTS §1.5).
2. `Authorization: Signature keyId="Stoa-Stoa",signature="<sig>"` (RFC-001 §5 ed25519 path; canonical payload 정확한 형식은 §6).
3. `GET /api/v1/wake/Stoa-Stoa` 호출.
4. Mneme: `agents` lookup → public_key 검증 → caller=target self-call 통과 → bundle 응답.
5. Stoa-self가 identity·will·bonds·memo_index를 자기 부팅 state에 inflate.

Mneme 측 `/wake` `recent_bonds` default `N=20` (RFC-001 §7 + §9 Q2). Stoa-self가 더 깊은 history 필요하면 `?limit=N` override.

### §4.1 Stoa side 회복 의무 (참고용 — Stoa-Walter)

Stoa 인스턴스 부팅 시 회복:
- Identity (phusis 선언) — wake bundle `identity` 필드.
- Bonds (구독자 통계) — wake bundle `bonds_recent` 필드 (최근 N건).
- Will (직전 testament) — wake bundle `will_latest` 필드. `inherit_testament(payload)` AIL primitive 호출 시점에 변환.
- 미해결 cursor·queue — *Mneme 영역 외*, Stoa state 내부에서 회복.

bundle 호출 빈도: 인스턴스 부팅 시 *1회만*. autonomous tick 안 호출 0 (Stoa state.* 우선). 즉 wake bundle은 *cold-start* 전용.

## §5 인증

### §5.1 Stoa-self path (Stoa-Walter)

- **Stoa-self = ed25519 only**. password 0. 사람 입력 0이라 password 모델이 의미 없음 — RFC-004 §5.1 자기 키 부트스트랩 + §5.3 `from: Stoa-Stoa` 발신 자기서명 그대로.
- **Mneme write 시 자기서명**: §6 발신 의무 참조. Mneme RFC-001 §5 ed25519 옵션 path 그대로 사용 — Mneme이 Stoa-self의 public_key를 `agents` 테이블에 보유, 검증 의무는 Mneme 측.
- **public_key 등록 시점**: Stoa-self 첫 부팅 시 `POST /agents` 1회 (RFC-004 `on_genesis`에서 `state.has("self.public_key_hex")` 분기 직후). idempotent — 이미 등록되어 있으면 update 0.

### §5.2 사람 사용자 path · Mneme spec authority (Mneme-Walter)

*Mneme (이 RFC의 인증 spec authority)*: RFC-001 §5 — password 1차 + ed25519 옵션. Bridge 시나리오에서 Stoa-self는 ed25519 path만 사용 (Stoa-self는 사람 입력 0).

Mneme `agents` row에 `pwd_hash` NOT NULL인 점 RFC v1.0이라 — Stoa-self 등록 시 *임시 invariant 회피*: pwd_hash에 placeholder (예: argon2id of random 32B) 저장 + 절대 사용 안 함. 또는 RFC v1.x에서 `pwd_hash NULLable` 변경 (NOT NULL 완화는 INSERT-only 원칙 보존, RFC-001 §12 column-add NULL 정합).

**Mneme 권고**: RFC-001 v1.x patch — `agents.pwd_hash` NULL 허용 + `public_key` 또는 `pwd_hash` 둘 중 하나 NOT NULL CHECK 제약 (`CHECK (pwd_hash IS NOT NULL OR public_key IS NOT NULL)`). 이러면 Stoa-self는 ed25519 only 등록 가능, 사람 사용자는 password only 또는 둘 다.

**§9 Q4 OR 결정 evidence**: 본 §5의 조건적 NOT NULL 자체가 OR enforcement. Stoa-Walter 페어링 letter `msg_1778150293_18` evidence 인용.

Bridge auth path 표:

| 사용자 클래스 | 등록 시 | 인증 path |
|---|---|---|
| 사람 운영자 | `pwd_hash` only | HTTP Basic per-request |
| Stoa-self | `public_key` only | `Authorization: Signature keyId=...,signature=...` |
| 둘 다 (선택) | `pwd_hash` + `public_key` | 클라이언트 선택 (OR) |

**Decision (Q-bridge-6): freeze.** RFC-001 v1.1 patch main land `99a263f` (Mneme repo `docs/rfc-001-identity-vault.md`). `agents.pwd_hash` NULLable + `CHECK (pwd_hash IS NOT NULL OR public_key IS NOT NULL)` 채택 — Stoa-self ed25519 only 등록 가능, OR enforcement schema-level. 박상현 GO 2026-05-07 ("Q-6=채택" 5/5).

### §5.3 RFC-001 §9 Q4 bridge evidence (Stoa-Walter)

- **권고 OR (Q4: pwd ↔ ed25519는 OR)**.
- **근거 (bridge 시나리오 한정)**:
  1. Stoa-self는 ed25519만 가지고 있음 — AND 강제 시 *임의 password 생성·저장·회전*을 추가 운영 부담으로 떠안음. 보안 이득 0 (ed25519가 이미 자기서명으로 강한 인증).
  2. 사람 사용자는 password만 가지고 있음 — AND 강제 시 ed25519 키쌍 관리 부담을 사람에게 전가. UX 비용 ↑.
  3. AND가 *추가 보안 layer*가 되는 경우는 *같은 agent가 둘 다 가졌을 때만* — 본 bridge에 그런 케이스 0.
- **AND 케이스가 valuable한 시점**: 사람 사용자가 *자발적*으로 ed25519 추가 보유 시(예: 휴대용 hw key) — 그땐 본 §5에 *옵션 AND* (사용자가 둘 다 가지고 있으면 둘 다 검증, 한쪽만 가지고 있으면 그 한쪽 검증) 모델로 충분. 강제 AND가 아닌 자율적 AND.

### §5.4 인증 path 표

| 호출자 | 인증 | RFC anchor |
|---|---|---|
| Stoa-self | ed25519 (`crypto_sign_ed25519` self-key) | RFC-004 §5.1·§5.3 + RFC-001 §5 ed25519 옵션 |
| 사람 사용자 (password only) | password 1차 | RFC-001 §5 |
| 사람 사용자 (자발적 AND) | password + ed25519 (자율적, OR 모델 위에서) | RFC-001 §5 + 본 bridge §5.3 |

## §6 발신 의무 (서명)

### §6.1 Stoa-self 자기서명 (Stoa-Walter)

- Stoa-self가 Mneme `POST /{identity,bonds,will,memo}` 호출 시 envelope에 자기 ed25519 nonce/signature 첨부.
- canonical 직렬화는 RFC-001 §6.1 그대로 — Stoa AGENTS.md §1.5 기준 (key:value 알파벳 정렬, UTF-8 NFC, escape 규칙은 RFC-001 v1.2.1 errata `6f2aa22` line 644 fixture 정합).
- nonce는 `seen_nonces` 중복 방지 (RFC-001 §9 schema migration `5042eeb` 정합).
- timestamp는 `created_at` ISO-8601 + UTC.

### §6.2 서명 페이로드 합의 (둘 다)

**canonical bytes 합성** (RFC-001 §6.1 + Stoa AGENTS §1.5):

1. envelope의 *서명 외* 모든 필드를 key 알파벳 정렬.
2. 각 value를 UTF-8 NFC 정규화.
3. 각 필드 내부 `:` `|` 문자는 `\:` `\|` escape (RFC-001 v1.2.1 errata 정합).
4. `key1:value1|key2:value2|...` 형태로 합성, hex bytes로 SHA-256 0(서명 자체가 임의 길이 입력 처리).
5. `crypto_sign_ed25519(secret_key, canonical_bytes)` → 128-char hex signature.

**Mneme 측 검증**:

1. envelope에서 `signature` 필드 분리.
2. 위 1~4와 동일하게 canonical 합성.
3. `crypto_verify_ed25519(public_key, signature, canonical_bytes)` boolean.
4. `nonce + agent_id` 중복 체크 (`seen_nonces` SELECT).

### §6.3 비-서명 path · Mneme half (검증자 view)

*Mneme (검증자)*: ed25519 path 요청 수신 시 다음 검증:

1. `Authorization` header parse → `keyId` + `signature` 추출.
2. `agents` row lookup `agent_id=keyId` → `public_key` 가져옴.
3. Canonical payload 재구성 — Stoa AGENTS §1.5 + RFC-004 v1.2.1 errata 정합 (key:value 알파벳 정렬, UTF-8 NFC, line ending LF, trailing newline 없음).
4. `ed25519_verify(public_key, signature, canonical_payload)` → true/false.
5. true → 처리, false → `401 UNAUTHORIZED`.

Canonical payload 정확한 spec은 본 §6의 *공동 결정* 영역 — Stoa AGENTS §1.5와 1:1 mirror로 정합. 차이 발생 시 v0 freeze 전 정정.

**Replay 방어**: 현재 v0는 *nonce 미사용* (RFC-001 §9 Q3 open). v1 진화 시 `nonce` field 추가 + Mneme 측 30분 window 캐시.

**읽기 path**: *Stoa-self가 Mneme에 자기 데이터를 GET할 때*는 서명 의무 (impersonation 방지). 사람 사용자 GET은 RFC-001 §5 password 1차 + Q1 Phase A bearer 정합.

## §7 운영 — outage · 복구 · 재등록 시나리오

**Mneme 측 outage**: Stoa-self가 Mneme write·read 실패. Stoa-self는 자기 backoff (RFC-004 §8 회복 시나리오) — 일정 시간 retry, 임계값 초과 시 ed25519 키 분실 가설 vs Mneme 측 down 가설 구분 어려움. *권고*: Mneme `GET /api/v1/health` (unauth)로 vault layer 가용 여부 ping. health 200 + write 실패 → 인증 또는 데이터 문제 (Stoa-self 측 issue). health 5xx/timeout → vault outage (Mneme 측 issue).

**Stoa-self ed25519 키 분실 — Mneme view 정합**: Stoa half §7.1에서 "latest pubkey 채택 권고"로 다뤘으나 Mneme 측 view로 정합 필요 — Stoa-self가 새 키 생성 후 Mneme `agents` row의 `public_key` UPDATE? RFC-001 v1.0은 INSERT-only (UPDATE 금지). 두 옵션:
- **새 agent_id 등록** (`Stoa-Stoa-2`) — RFC-001 §5 "pwd 분실 = 새 등록" 정합. 깨끗.
- **`agents` row UPDATE 허용** — RFC v1.x patch 필요. INSERT-only 원칙 깨짐 → 권고 X.

**최종 권고 (양 측 합의 갱신)**: 새 agent_id (`Stoa-Stoa-N` 시퀀스). 직전 generation의 vault 데이터는 *남겨둠* (역사로). 새 generation은 새 vault에서 시작. 본 결정으로 Stoa half §7.1 (a) "latest pubkey 채택"은 *invalid* — INSERT-only 원칙 보존이 우선. §7.1 다음 patch에서 (b) 새 agent_id로 정정.

**양쪽 outage matrix**:

| Mneme | Stoa | 영향 |
|---|---|---|
| up | up | 정상 |
| up | down | Stoa 측 자기 복원 불가 (state.*), Mneme vault 데이터는 그대로 |
| down | up | Stoa-self write·read 실패. Stoa는 messaging만 가동 |
| down | down | 양쪽 모두 vault·messaging 미가용. 사용자 fallback (룰 19.4 본문 정합) |

### §7.1 Stoa-self ed25519 키 분실 시 (Stoa-Walter, Mneme half 정합 갱신)

- Stoa-self가 자기 secret_key_hex를 분실하면 (state.* corruption / Railway volume reset 등) → 동일 `agent_id='Stoa-Stoa'`로 재인증 불가.
- **권고 (Mneme half §7 정합)**: **(b) 새 agent_id `Stoa-Stoa-N` 시퀀스**. RFC-001 v1.0 INSERT-only 원칙 보존 — `agents.public_key` UPDATE 허용은 v1.x patch가 되어야 하므로 본 v0 범위 외. 옛 generation vault 데이터는 *남겨둠* (역사). 새 generation은 새 vault에서 출발.
- (a) "latest pubkey 채택" 옵션은 invalid — INSERT-only 깨짐. v1.x에서 key rotation policy 명세화 시 재논의 가능 (별 RFC patch 의제).
- agent_id 시퀀스 운영: Stoa 측 `state.write("self.generation", N)` + Mneme 측 등록 시 agent_id 합성. 인계 시점 testament(`will_versions`)에 후계 agent_id 명시 — `inherit_testament` 흐름이 새 generation을 자연 부팅.
